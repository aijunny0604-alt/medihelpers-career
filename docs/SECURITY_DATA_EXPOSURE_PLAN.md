# SECURITY · 공고 급여·병원정보 유출 방어 계획

기준일: 2026-07-18
관련 문서: `ARCHITECTURE.md`, `API.md`, `STATUS.md`, `AUTH.md`, `DB.md`
심각도: **High (구조적)** · 현재 실제 데이터는 데모라 즉시 위험은 낮으나, **실제 공고 투입 전 반드시 선행**해야 함.

## 문제 요약 (보안 감사 결과)

공고 상세의 "의사 인증 후 공개 / 멤버십 구독 후 공개" 잠금이 **가짜 잠금**이다. 서버는 전체 데이터를 다 내려주고, 화면에서 글자만 가린다.

- **근원**: 공고·인재 데이터가 정적 파일 `src/data.js`에 통째로 담겨 클라이언트 번들로 전송됨.
- **노출 경로**:
  - 개발자도구 → Sources에서 번들 JS의 `data.js` 열람
  - `/qa-preview`에서 "멤버십 이용 회원" 상태 클릭 → localStorage 플래그만으로 전 잠금 해제 (네트워크 요청 0건)
- **노출 필드**: 모든 공고의 `pay`(급여)·`workHours`·`daysOff`, 데모 공고의 `businessNumber`·`representative`·`fullAddress`·`doctorCount`·`equipment` 등 병원 식별정보.
- **연관**: **매칭 리포트(`MatchingReportPage`)도 동일 공고 데이터 기반**이라 같은 뿌리 문제. 데이터를 서버로 옮기면 매칭 리포트도 함께 안전해진다.
- **부가 문제**: rate limit·봇 차단·서버 페이지네이션 전무 → 대량 스크래핑 무방비.

## 판정 표 (클라이언트 마스킹 vs 서버 미제공)

| 데이터 | 현재 방어 | 목표 |
|---|---|---|
| 공고 급여·근무조건 | ❌ 클라이언트 마스킹(가짜) | ✅ 서버가 비인증 응답에서 필드 제외 |
| 병원 식별정보(사업자·대표·주소) | ❌ 정적 노출 | ✅ 서버 인증 응답에만 포함 |
| 후보 실명·연락처 | ✅ 서버 admin 전용 + 감사로그 | 유지 |
| 관리자 콘텐츠 visibility | ✅ 서버 필터 | 유지 |
| 인재 익명화(김○○) | ✅ 원천 익명(실명 데이터 없음) | 유지 |

## 목표

1. 공고 상세의 급여·상세근무조건·병원 식별정보를 **비인증/비멤버십 응답에는 필드 자체를 넣지 않는다**(서버 필터).
2. 정적 `data.js` 공고를 서버 원장(D1)로 이전하고, 공개 목록 API와 상세 API를 분리한다.
3. 공개 API에 rate limit·서버 페이지네이션을 도입한다.

## 데이터 분류 (공개 / 인증 / 멤버십)

| 구분 | 필드 | 비인증 | 의사 인증 | 멤버십 |
|---|---|:--:|:--:|:--:|
| 공개 요약 | hospital, title, dept, region, type, badge, deadline, summary(요약), benefits(대표) | ✅ | ✅ | ✅ |
| 기본 조건 | pay(급여), workHours, daysOff | ❌ | ✅ | ✅ |
| 상세 조건 | 보수구조·실근무표·진료강도·입사판단(`doctorDecisionGroups`) | ❌ | ❌ | ✅ |
| 병원 식별 | businessNumber, representative, fullAddress, doctorCount, equipment, beds | ❌ | 부분 | ✅ |

> ※ 경계는 초안. BILLING/AUTH 원칙("의사 인증 시 핵심정보 무료")과 맞춰 확정 필요.

## 구현 단계 (안전하게 순차)

1. **M1 · 상세 API 분리** — `GET /api/jobs/:id`를 신설(또는 site-operations 확장)해 서버가 요청자 역할(비인증/doctor/membership)에 따라 필드를 **골라 내려준다**. 클라이언트는 받은 것만 렌더(마스킹 로직 제거).
2. **M2 · 목록 API + 페이지네이션** — 공개 목록을 서버 페이지네이션으로 제한. 정적 `data.js`는 시드/폴백으로만.
3. **M3 · 정적 데이터 이전** — `data.js` 공고를 D1 `job_posts`로 이전(단일 원장화, `STATUS.md` 이미 인지한 과제).
4. **M4 · 스크래핑 방어** — Worker 레벨 IP/세션 rate limit, 공개 목록 크기 서버 강제, 봇 차단.
5. **M5 · 매칭 리포트 연동** — 매칭 리포트도 상세 API 기반으로 전환해 동일 필터 적용.

## 즉시 가능한 완화(선택)

서버 이전 전이라도, 실제 공고 투입 전 `data.js`에서 병원 식별 민감필드(사업자번호·대표명·전체주소)를 빼거나 빈값 처리하면 표면 노출은 줄일 수 있다. 단, 급여 잠금은 서버 분리 전까지 근본 해결 불가.

## 리스크·주의

- 서버 런타임이 OpenAI Sites Worker(D1)라, 엔드포인트 추가 시 배포 파이프라인(`scripts/package-sites.mjs`) 수정 필요.
- 필드 필터 경계를 잘못 잡으면 "의사 인증했는데 급여가 안 보임" 같은 UX 회귀 → M1에서 QA 미리보기 5개 역할로 반드시 검증.
- 데이터 이전 중 공고 노출 누락 방지(마이그레이션 검증).

## 열린 질문

- 급여를 "비인증 완전 비공개"로 할지, "범위만 노출(예: 월 1,000만원대)"로 할지 — 마케팅상 완전 비공개는 유입 저하 우려.
- 병원명·로고는 공개 유지(브랜드 노출 상품이므로) 확정.
