# 의사 초빙 정보란 + 회원 전용 공개 기획

기준일: 2026-07-18
관련 문서: `AUTH.md`, `BILLING.md`, `SECURITY_DATA_EXPOSURE_PLAN.md`, `ARCHITECTURE.md`

## 배경

기존 medihelpers.co.kr에는 **"의사 초빙 정보란"** 게시판이 있었다. 대표(이형석)·admin·관리자만 글을 올리고("[부산] 소화기 내과 선생님 초빙!" 등), 병원 의뢰를 받아 아빠(헤드헌터)가 직접 큐레이션해 공고를 게시하던 **헤드헌팅 사업의 핵심 도구**다. 배너 문구: "구인광고 없이 비밀리에 의료인 초빙 = 의사헤드헌팅 사업부". 새 사이트(medihelpers-career)에는 이에 대응하는 전용 게시판이 없다.

## 세 갈래 공고 구조 (혼동 방지 — 확정)

| 구분 | 올리는 주체 | 열람 | 급여·조건 | 성격 |
|---|---|---|---|---|
| **채용정보(광고)** `/jobs` | 병원(유료) | 누구나 | **공개** | 널리 알리는 광고 |
| **의사 초빙 정보란**(신규) | **아빠(관리자)만** | **회원 이상** | 회원엔 공개 | 아빠가 큐레이션한 초빙 |
| 비공개 헤드헌팅 포지션 | 아빠 | 상담자 | 상담 후 | badge='비공개' |
| **인재정보** `/talent` | (의사 구직측) | **회원 이상** | 익명·동의 후 실명 | 구직 인재풀 |

> ※ 초빙 정보란(병원→의사 구인)과 인재정보(의사→병원 구직)는 **대체 관계가 아니라 짝**이다. 둘 다 있어야 아빠가 양방향 매칭 가능. 초빙을 인재정보로 대체하면 "병원 구인"이 사라진다.

## 핵심 원칙: 회원 전용 열람 (경쟁 헤드헌터 방어)

사용자 요구: **초빙 정보란·인재정보 글은 회원 이상만 열람. 비회원·비공개회원은 내용을 못 본다.**

- 비회원/비로그인 → 목록의 제목·지역 정도만(또는 블러) + "로그인 후 열람" 유도. 상세 내용 차단.
- 로그인 회원(의사) → 전체 열람.
- 이 게이트는 **서버에서 강제**해야 한다(클라이언트 마스킹 금지). 아빠가 올린 초빙 상세·인재 상세는 비인증 API 응답에 넣지 않는다. [[SECURITY_DATA_EXPOSURE_PLAN]]

### 왜 이게 방어가 되나
- 경쟁 헤드헌터가 **가입·인증 없이는 목록을 못 봄** → 진입장벽.
- 인재정보는 이미 익명화(김○○, 근무기관·연락처 잠금, "헤드헌터 상담·동의 후 공개")되어 있어, 회원이 봐도 **직접 접촉 불가 → 반드시 아빠 경유**. 실명·연락처는 서버가 admin에게만 제공(보안 감사 확인).
- 즉 "정보는 병원 유입용으로 노출하되, 실제 연결은 아빠 독점" 구조 유지.

## 표시 형태 (권장: 게시판+카드 하이브리드)

- **목록**: 게시판형(번호·제목·지역·등록일) — 아빠가 익숙하고 많은 공고를 빠르게 관리. 기존 운영방식 계승.
- **상세**: 새 사이트 상세 카드 스타일 재사용(모집개요·병원정보). 디자인 일관성 유지.
- 상단 배너에 "의사 헤드헌팅 사업부 · 회원 전용" 명시.

## 기술 접근 (기존 구조 재사용)

- 아빠 등록: 관리자 콘솔의 콘텐츠 CRUD(`admin_content_records`, content_type 확장 예: `headhunting_post`) 재사용. 공개 가입 대상 아님(관리자만).
- 열람 게이트: `/api/site-operations`의 visibility 화이트리스트에 회원 조건 추가. 비회원엔 상세 payload 미제공.
- 라우팅: `/headhunting-board`(또는 기존 `/headhunting` 확장) 신설, navItems 추가.
- 급여 로직: 이미 `restricted = badge==='비공개'`로 수정됨(광고는 공개). 초빙 정보란은 "회원 게이트"가 별도 축.

## 마일스톤

1. **M1 설계** — 콘텐츠 타입/스키마, 회원 게이트 서버 규칙, 라우트·메뉴 확정.
2. **M2 목록/상세** — 게시판 목록 + 상세 화면, 비회원 잠금.
3. **M3 관리자 작성** — 아빠 전용 등록/수정 흐름(관리자 콘솔 연동).
4. **M4 인재정보 회원 게이트** — 인재정보 상세도 회원 전용으로 서버 강제.

## 확정 사항 (2026-07-18 사용자 결정)

### 1. 등록 주체 = 아빠(관리자)만
- **인재정보(`/talent`)·초빙 정보란 글은 의사·병원이 직접 못 올린다. 아빠(관리자)만 등록.**
- 현재 구조가 이미 그러함: `talent_profile` 등 콘텐츠는 `admin_content_records`에 관리자 콘솔(`adminIdentity` 게이트)로만 등록. 의사는 이력서(`/resume`)·구직상담(`/request`)만 하고, 아빠가 검토 후 인재 프로필로 큐레이션해 등록.
- 이유: 아빠가 게이트키퍼 → 가짜 프로필·경쟁자 침투 차단, 품질 관리.

### 2. 인재정보(`/talent`) 열람 게이트 = "비회원 흐리게, 열람은 병원회원"
- **현재 상태**: 목록은 누구나 열람(비회원도 익명 프로필 봄), 실명만 병원/관리자 전용(`main.jsx:1478~1486`, 서버 role 확인). → 목록 접근 제한 없음.
- **목표**: 비회원/일반회원에겐 인재 목록을 **블러 또는 일부만** 노출 + "병원 회원 가입 후 전체 열람" 유도. 전체 열람은 **병원 회원**부터.
- 완전 차단이 아니라 블러로 하는 이유: 비회원 병원이 "볼 게 있네" 맛보기 → 가입 동기 유지(병원 유치)와 방어를 동시에.
- **서버 강제**: 블러 대상 상세 데이터는 비병원 응답에 아예 안 실어야 함(클라 마스킹 금지). [[SECURITY_DATA_EXPOSURE_PLAN]]
