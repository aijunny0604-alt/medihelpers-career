# 현재 구현 상태

기준일: 2026-07-17

## 운영 중

- React·Vite SPA를 OpenAI Sites에 배포
- D1 스키마와 런타임 API를 `scripts/package-sites.mjs`에서 패키징
- 홈·채용정보 공고 카드 디자인과 노출 순서 통일
- 의사 초빙정보, 익명 인재정보, 헤드헌팅 상담, 의료인 채용정보
- 일반 회원·병원 회원 가입 화면과 역할별 마이페이지
- 상담 접수 D1 저장, 관리자 상담함, 이메일·문자 알림 시도 결과 기록
- 관리자 통합 콘솔, 카테고리·기능·콘텐츠·회원·결제 요청·감사 로그
- 채용 CRM의 의뢰→조건 확인→후보 탐색→동의→제안→면접→협상→입사 단계

## 부분 구현

| 영역 | 현재 동작 | 운영 전 남은 작업 |
|---|---|---|
| 회원가입 | Sites 인증 사용자와 역할·동의 기록 구조 | 승인된 약관 버전, 본인확인, 면허·병원 인증 |
| 결제 | 서버 상품 원장, 주문·거래·영수증·환불 요청 테이블, 관리자 수동 상태관리 | PG 승인 API, 웹훅 검증, 자동 취소·환불·증빙 |
| 공고·인재 관리 | 관리자 D1 콘텐츠를 공개 API로 화면에 병합 | 정적 샘플 전체를 D1로 이전하고 단일 원장화 |
| 상담 | 로그인 확인, D1 저장, 관리자 처리상태·메모 | 계정 ID 직접 연결, 알림 재시도·담당자 배정 |
| 이미지 | 프론트엔드 선택·미리보기·규격 안내 | R2 업로드, 소유권·검수·삭제 정책 |
| 후보 개인정보 | 익명 표시와 동의 기반 공개 원칙·CRM 스키마 | 별도 비공개 저장소, 병원별 동의 API, 열람 감사 |

## 이번 최신화에서 확인·수정

- 상담 제출 시 가입된 계정의 역할과 상담 유형이 다르면 서버가 거부합니다.
- 상담 내역의 회원 연결 기준 이메일은 사용자가 임의 입력한 값이 아니라 인증된 계정 이메일로 고정했습니다.
- 손상된 상담·관리 콘텐츠 JSON 한 건 때문에 마이페이지나 관리자 콘솔 전체가 열리지 않도록 안전 파싱을 적용했습니다.
- 공개 카테고리 API의 캐시 헤더가 실제 응답에 반영되도록 응답 헬퍼를 수정했습니다.

## 알려진 핵심 차이

관리자 화면의 콘텐츠 건수는 D1에 등록된 운영 콘텐츠만 집계합니다. 공개 채용정보 화면은 D1 운영 콘텐츠와 `src/data.js`의 시연용 정적 데이터를 합쳐 보여주므로 현재 두 화면의 건수는 같지 않습니다. 운영 전 정적 데이터를 D1로 이전한 뒤 관리자 DB를 유일한 데이터 원장으로 바꿔야 합니다.

## 출시 차단 조건

- 약관·개인정보처리방침·제3자 제공·환불정책 법률 검토
- `ADMIN_EMAILS`, `ACCOUNT_HASH_SECRET`, 가입 기능 플래그 운영값 검증
- 면허·병원 담당자 인증
- PG 계약과 승인·웹훅·취소·환불 통합 테스트
- R2 파일 업로드와 악성 파일 검사
- 정적 샘플의 운영 DB 이전
