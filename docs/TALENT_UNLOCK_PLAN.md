# 인재 이력서 열람권 수익모델 기획 (병원 결제 → 연락처·이력서 공개)

기준일: 2026-07-18
관련: `BILLING.md`, `HEADHUNTING_BOARD_PLAN.md`, `RESUME_PIPELINE_PLAN.md`, `SECURITY_DATA_EXPOSURE_PLAN.md`
참고: 기존 medihelpers.co.kr 실제 방식(구직 프로필 공개 + 연락처·이력서는 열람권 결제)

## 핵심 사업 모델 (사용자 확정)

기존 메디헬퍼스의 진짜 수익 = **병원이 인재(구직 의료인)의 연락처·이력서를 보려면 "이력서 열람권"을 결제**한다.

```
1. 의료인(의사·간호사 등) 회원가입 → 이력서 등록 → "구직희망" 등록
2. 인재정보(/talent)에 익명 노출: 이름 이**, 희망지역·직종·경력 등 기본정보는 공개
3. 병원이 관심 → 이력서 열람권 결제 → 연락처·이력서 상세·자기소개·첨부 열람
4. 병원↔의사 매칭 (또는 아빠 헤드헌터 개입)
```

이것이 **아빠 사업의 주 수익원**(성공보수와 별개로, 열람권 판매). 광고비·멤버십보다 이게 핵심.

## 공개 3단계 (기존 메디헬퍼스 기준)

| 정보 | 비회원 | 병원 회원(열람 전) | 열람권 결제 후 |
|---|---|---|---|
| 이름 | 이**(익명) | 이**(익명) | 실명 |
| 희망지역·직종·경력·희망급여·현재상태 | 공개 | 공개 | 공개 |
| **전화번호·핸드폰·이메일·주소** | ❌ | ❌ "열람신청" | ✅ 공개 |
| **수행프로젝트·자기소개·이력서 파일** | ❌ | ❌ "이력서 열람 신청" | ✅ 공개 |

※ 우리는 파일 업로드 제거(우리 이력서 전용)했으므로 "이력서 파일" 대신 구조화 이력서 상세로 대체.

## 열람권 상품 설계 (초안)

- **건별 열람권**: 후보 1명 이력서·연락처 열람 (예: 건당 N원)
- **기간 이용권**: 월정액으로 기간 내 무제한/N명 열람 (예: 월정액)
- (기존 medihelpers처럼 "입사지원 요청"·"면접제의"·"인재정보 스크랩" 버튼도 고려)
- `payment_orders.product_type`에 `talent_search` 추가(스키마 enum엔 이미 관련 흔적 있음, 확인 필요), `paymentProductCatalog`에 상품 등록.

## 서버 구현 (열람권 = 진짜 보안 게이트)

⚠️ **핵심**: 실명·연락처·이력서 상세는 **서버가 결제한 병원에게만** 준다(클라 마스킹 아님). [[SECURITY_DATA_EXPOSURE_PLAN]]
- 새 테이블 `talent_unlocks(hospital_account_id, talent_id, order_id, expires_at)` 등으로 열람 권한 기록.
- `/api/talent/:id` 상세 API: 요청 병원이 해당 인재 열람권 보유 시에만 실명·연락처·상세 반환. 아니면 익명·기본정보만.
- 열람 시 감사 로그(`access_audit_logs`) — 누가 언제 누구를 열람했는지.
- 후보 동의(identityConsent) 여전히 필요: 열람권만으로 부족, 후보가 공개 동의한 범위 내.

## 결제 흐름 (이니시스/가상결제 재사용)

기존 결제 시스템([[BILLING_INICIS_PLAN]], 가상결제 테스트모드) 재사용:
```
병원이 인재 "열람권 구매" → POST /api/payment-orders (product=열람권)
→ 승인(가상 or 이니시스) → paid → talent_unlocks에 권한 기록
→ /talent/:id가 실명·연락처·상세 공개
```

## 멤버십과의 관계 (정리)

- **의사 급여 열람**: 로그인만 하면 무료(메디게이트 방식, 이미 구현) — 의사 유입용.
- **병원 인재 열람권**: 결제 필요 — **주 수익**.
- 기존 "의사 멤버십(1.9만/3.9만)"은 부가(알림·분석)로 재정의하거나 병원 열람권 중심으로 재편 검토.

## 마일스톤

1. **M1** 열람권 상품·결제(payment_orders 재사용) + talent_unlocks 테이블
2. **M2** /api/talent/:id 상세 API(열람권+동의 검증, 서버 필터)
3. **M3** 인재정보 UI: "열람신청/이력서 열람" 버튼 → 결제 → 상세 공개
4. **M4** 이력서 → 인재정보 연동(아빠 큐레이션, 동의 기반)
5. **M5** 입사지원요청·면접제의·스크랩 등 부가 액션
