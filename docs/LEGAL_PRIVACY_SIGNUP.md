# 회원가입·개인정보·소셜 로그인 법무/보안 기준

## 문서 지위

- 기준일: 2026-07-16, 대한민국에서 운영되는 메디헬퍼스 웹서비스를 전제로 한 출시 전 검토 초안입니다.
- 이 문서는 제품·개발·운영팀의 구현 기준이며 법률 자문서가 아닙니다.
- 실제 공개 전에는 사업자 정보, 서비스 방식, 수탁사 계약, 데이터 보관 위치를 확정한 뒤 대한민국 변호사 또는 개인정보 보호책임자의 최종 검토를 받아야 합니다.
- `[확정 필요]`가 남아 있으면 공개 회원가입, 의료인 서류 수집, 병원 정보 제공, 유료 결제를 열지 않습니다.

## 핵심 결론

1. 서비스 이용약관 동의와 개인정보 처리는 별개입니다.
2. 계정 생성·보안 등 계약 이행에 필요한 최소정보는 개인정보 보호법 제15조제1항의 적법 근거를 검토하고 처리방침에 공개합니다. 필요하지 않은 동의를 관행적으로 필수화하지 않습니다.
3. 동의를 근거로 처리할 때는 수집·이용, 제3자 제공, 민감정보, 고유식별정보, 홍보·판매 권유를 서로 구분하고 명확히 표시합니다.
4. 마케팅 수신 동의는 선택이며 이메일·문자·카카오 등 채널별로 기록하고 언제든 철회할 수 있어야 합니다.
5. 의료인의 실명·연락처·이력서는 회원가입 때 포괄적으로 병원 제공 동의를 받지 않습니다. 실제 소개 요청마다 제공받는 병원, 목적, 항목, 보유기간을 보여주고 별도 동의를 받습니다.
6. 면허 확인에 주민등록번호를 수집하지 않습니다. 주민등록번호는 단순 동의만으로 처리할 수 없으며 구체적인 법령 근거 등이 있을 때만 가능합니다.
7. Google·Kakao·Naver 로그인은 인증 수단일 뿐 메디헬퍼스의 개인정보 처리 고지나 동의를 대신하지 않습니다.
8. Supabase, 소셜 로그인 사업자, 이메일·문자 사업자 등이 국외에서 개인정보를 처리·보관한다면 실제 계약과 리전을 기준으로 국외 이전의 적법 근거와 고지 사항을 확정해야 합니다.
9. 직업정보 제공인지 유료 직업소개인지에 따라 신고·등록 의무가 달라질 수 있습니다. 회원가입 공개 전 실제 사업 흐름을 관할 기관 또는 전문가와 확인합니다.

## 가입 화면 권장 구조

### 1. 가입 전 역할 선택

- 의료인
- 병원·의료기관 담당자
- 헤드헌터·운영자 계정은 공개 가입을 막고 관리자 초대로만 생성

역할 선택은 권한을 자동 부여하는 수단이 아닙니다. 의료인·병원 권한은 서버 검증 후 별도로 부여합니다.

### 2. 소셜 로그인 버튼

> 간편 로그인은 본인 인증과 계정 생성을 위한 수단입니다. 메디헬퍼스는 선택한 로그인 사업자로부터 서비스에 필요한 최소 정보만 제공받습니다. 제공 항목은 로그인 사업자의 동의 화면에서 확인할 수 있습니다.

수집 범위:

| 구분 | 기본 처리 | 원칙 |
| --- | --- | --- |
| 로그인 사업자 | Google/Kakao/Naver | 계정 식별과 장애 대응에 필요 |
| 사업자 사용자 식별자 | 필수 | 이메일 대신 안정적인 계정 키로 사용 |

| 이름/닉네임 | 최소 선택 | 서비스 표시 또는 본인확인에 필요할 때만 |
| 프로필 사진 | 기본 미수집 | 명확한 기능이 생기기 전 요청하지 않음 |
| 친구·연락처 목록 | 수집 금지 | 회원가입에 불필요 |

서로 다른 로그인 사업자의 이메일이 같다는 이유만으로 계정을 자동 연결하지 않습니다. 로그인된 계정에서 재인증한 경우에만 연결하고 연결·해제 이력을 남깁니다.

- `[필수] 서비스 이용약관 동의` — 전문 보기

`전체 동의`는 편의 기능으로 둘 수 있지만 다음을 지킵니다.

- 선택 동의가 필수처럼 보이지 않게 구분합니다.
- 개별 체크 상태와 전문 링크를 바로 확인할 수 있게 합니다.
- 기본값은 모두 체크 해제입니다.
- 선택 동의를 거부해도 가입·핵심 서비스 이용에 불이익을 주지 않습니다.

- 담당자 개인 연락처 대신 기관 업무 연락처를 우선합니다.
- 병원 관리자 권한은 서버의 승인 절차 후 부여합니다.

- 병원이 내려받은 정보의 파기 확인과 처리 책임을 계약에 포함합니다.

### I. 국외 이전 고지/동의 템플릿

Supabase 프로젝트 리전, 소셜 로그인, 분석, 이메일·문자 도구와 계약을 확정한 뒤 실제 값으로 채웁니다.

| 이전받는 자와 연락처 | 이전 국가 | 이전 항목 | 목적 | 시기·방법 | 보유·이용 기간 | 거부 방법·효과 |
| --- | --- | --- | --- | --- | --- | --- |
| `[법인명/연락처 확정]` | `[실제 처리 국가]` | `[실제 항목]` | 인증, 데이터 보관, 보안 등 `[확정]` | 가입·로그인 시 암호화 통신망 전송 등 `[확정]` | `[계약과 삭제 정책 기준]` | `[거부/철회 방법 및 서비스 영향]` |

주의:

- 브랜드명만 보고 법인명·국가·보유기간을 추측하지 않습니다.
- 개인정보 처리 위탁과 제3자 제공을 구분합니다.
- 계약 이행에 필요한 국외 처리위탁·보관 근거를 사용하는지, 별도 동의를 받을지는 실제 구조와 법률 검토로 결정합니다.
- 처리방침 공개만으로 가능한 경우에도 법정 고지 항목을 모두 적고, 사업자 변경을 정기 점검합니다.
- 사용자가 국외 이전을 거부할 수 있는 대체 가입 수단이 필요한지 함께 결정합니다.

## 개인정보 처리방침 필수 구성 체크

- 개인정보처리자 실제 상호·대표 연락처 `[확정 필요]`
- 개인정보 처리 목적, 항목, 보유기간과 각각의 법적 근거
- 만 14세 미만 아동 처리 여부
- 제3자 제공 현황 또는 해당 없음
- 처리 위탁 업체, 업무 내용과 재위탁 관리
- 국외 이전 내역과 적법 근거
- 파기 절차·방법, 분리 보관
- 정보주체와 법정대리인의 열람·정정·삭제·처리정지·동의철회 방법
- 개인정보 보호책임자와 고충 처리 창구 `[확정 필요]`
- 안전성 확보조치
- 자동 수집 장치와 거부 방법
- 자동화된 결정이 있다면 기준·절차·거부 및 설명 요구 방법
- 처리방침 변경 이력, 시행일과 이전 버전 보기
- 개인정보 침해 신고·분쟁조정 기관 안내

개인정보 처리방침은 가입 화면에서만 링크하지 말고 모든 페이지의 푸터와 계정 설정에서 쉽게 접근할 수 있게 공개합니다.

## 데이터와 동의 기록 규칙

`consents` 또는 동등한 테이블에 다음을 기록합니다.

- `user_id`
- `consent_type`: terms, privacy_consent, privacy_notice_ack, marketing_email, marketing_sms, marketing_kakao, third_party_job_intro, sensitive_data, overseas_transfer 등
- `legal_basis`: consent, contract, legal_obligation, legitimate_interest_review 등
- `document_version`과 문서 해시
- 동의 또는 확인 상태, 일시, 철회 일시
- 화면·앱 버전, 언어, 가입 경로
- 제3자 제공이면 수신 병원 ID, 제공 항목 스냅샷, 목적, 보유기간
- 국외 이전이면 이전받는 자·국가·항목·목적·기간 스냅샷

금지 사항:

- 개인정보 처리방침을 `terms_accepted=true` 하나로 합치기
- 현재 문구만 저장하고 사용자가 동의한 당시 버전을 보존하지 않기
- 선택 동의를 역할이나 유료 권한 부여 조건으로 사용하기
- 클라이언트가 보낸 역할·인증 완료 값을 그대로 신뢰하기

## 보안 출시 기준

### 인증·세션

- OAuth Authorization Code + PKCE와 정확한 redirect URL 허용목록 사용
- OAuth client secret, Supabase service role key 등 비밀값을 브라우저 번들에 포함하지 않음
- 관리자·헤드헌터 계정 MFA 의무화
- 세션 만료, 전체 로그아웃, 탈퇴 시 토큰 폐기와 소셜 계정 연결 해제 절차 제공
- 로그인·가입·비밀번호 재설정·인증 코드에 속도 제한과 이상 징후 탐지 적용

### 권한·데이터

- 모든 개인정보 테이블에 서버 측 권한 검사와 Supabase RLS 적용
- 사용자 공개 프로필과 연락처·서류·감사 로그를 물리적 또는 논리적으로 분리
- `Doctor`, `Hospital`, `Admin` 권한은 서버 승인 함수만 변경 가능
- 관리자 조회, 다운로드, 후보 소개, 권한 변경을 감사 로그로 남김
- 저장·전송 구간 암호화, 백업 복구 테스트, 키 정기 교체 절차 마련
- 업로드 파일 확장자·MIME·크기 검사, 악성 파일 검사, 임의 파일명, 비공개 버킷 사용
- 운영·개발·분석 환경에 실제 개인정보를 복제하지 않음

### 운영

- 내부 관리계획, 접근권한 검토, 수탁사 관리, 사고 대응과 통지 절차 문서화
- 개인정보 보유기간 만료 자동 파기 작업과 실패 알림 운영
- 퇴사자·외주 인력 권한 즉시 회수
- 최소 연 1회 권한·보유기간·수탁사·국외 이전·처리방침 정합성 검토
- 유출 의심 시 증거 보존, 영향 범위 확인, 법정 신고·통지 기한을 담당자가 즉시 판단할 수 있는 대응표 마련
