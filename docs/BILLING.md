# BILLING

## 수익 구조 (2026-07-22 갱신 — 의사 멤버십 폐지)

**유료 상품은 두 가지뿐이다. 둘 다 병원이 결제한다. 의사·의료인은 전부 무료.**

1. **병원 인재 이력서 열람권** — 주 수익. 병원이 인재 연락처·이력서 상세를 열람하려면 결제(1명 5,000원 / 5명팩 20,000원=건당 4,000원, 30일). 단가는 운영 데이터로 재조정. [[TALENT_UNLOCK_PLAN]]
2. **병원 의사 초빙공고 광고** (베이직 59,000 / 추천 149,000 / 집중 299,000원)

그 외(계약 기반, 사이트 밖):
- 병원 의사 헤드헌팅 성공보수 (아빠 직접 매칭, 오프라인 계약)

### 정책 원칙
- **의사 대상 유료 멤버십은 폐지됐다(2026-07-22).** 예전의 '커리어 체크(19,000원)·커리어 컨시어지(39,000원)'는 상품 카탈로그·화면·라우트에서 모두 제거했다. 의사 회원을 최대한 모아 인재풀을 키우는 것이 병원 수익보다 우선이라는 판단.
- **의사 급여·공고 상세는 로그인 회원이면 무료 열람.**
- **열람권은 병원 회원만 구매할 수 있다.** 열람 권한은 병원에만 부여되므로, 의사가 결제해도 쓸 수 없는 상품이 된다(서버에서 403으로 차단).
- **후보 개인정보(실명·연락처)는 상품으로 직접 판매하지 않는다.** 병원이 사는 것은 "열람권"이며, 후보 동의 범위에서만 열람. 인재정보 목록은 항상 익명.
- **병원 채용공고는 무료 게시 불가** — 광고 상품 결제로만 게시(서버 `job_create` 403 차단).
- 헤드헌터 상담은 무료.

## 현재 구현 (2026-07-22)

- 서버 상품 카탈로그(**광고 3종 + 인재 열람권 2종**, 총 5종)로 상품명·금액 확정. 멤버십 2종은 제거됨
- 역할별 구매 가능 상품 검사(광고=병원, 열람권=병원), 주문 원장(주문·거래·영수증·환불·웹훅 스키마)
- **가상(테스트) 결제 모드**: 이니시스 키 미설정 시 결제 흐름을 시뮬레이션(주문→paid→거래기록 실제 작동, 실제 청구만 없음). 키 설정 시 자동으로 실제 이니시스 승인으로 전환.
- **이니시스 승인 API 골격**: `POST /api/payment-approve`(금액 위변조 검증), 웹표준결제 파라미터 생성. INICIS_MID/SIGN_KEY 설정 시 작동.
- **결제 라이프사이클 자동화**: 광고 결제완료→노출기간 부여, 환불/취소→노출 회수. 병원 마이페이지 '내 공고' 이용현황, 관리자 결제화면 노출기간 표시.
- **인재 열람권**: 병원 결제→`talent_unlocks` 권한 기록→`GET /api/talent-detail/:id`가 그 병원에만 연락처·이력서 상세 제공(감사 로그). 열람 없으면 잠금+구매버튼.
- 회원 마이페이지 주문 이력(실제 집계), 관리자 통합 콘솔·감사 로그.

## 아직 실제 결제가 아닌 부분

이니시스 **연동 코드는 완료**됐고(결제창 호출·승인 API·금액 검증·멱등성·SSRF 방어), 남은 것은 계약·설정이다.

- **이니시스 키 미발급** → `INICIS_MID`/`INICIS_SIGN_KEY`/`SITE_ORIGIN` 미설정 상태라 가상(테스트) 결제로 동작
- **OpenAI Sites에서는 실결제 불가**(약관 §2.5/3.3 PCI 금지) → Cloudflare 이전 필요 [[CLOUDFLARE_MIGRATION]]
- 결제 웹훅 수신·서명 검증
- 카드 취소와 실제 환불 실행(현재는 관리자 수동 기록)
- 세금계산서·현금영수증 자동 발행

따라서 현재 `paid`, `cancelled`, `refund requested` 상태는 관리자 운영 시연과 수동 기록을 위한 값입니다. PG 거래 결과로 간주하면 안 됩니다.

⚠️ **열람권 5명팩 수량 미구현**: `talent-unlock-pack`(20,000원)을 5명으로 팔지만 서버는 `unlockCount`를 읽지 않아 **1명만 발급**된다. 실결제 전에 반드시 수정해야 하는 과금 정확성 문제.

## 운영 결제 전 필수 규칙

- 주문 금액과 상품 조건은 주문 생성 시점 스냅샷으로 보존합니다.
- PG 승인 금액과 서버 주문 금액이 다르면 승인하지 않습니다.
- 웹훅은 공급자 이벤트 ID를 멱등키로 사용합니다.
- 환불 합계는 원 결제액을 넘을 수 없습니다.
- 결제·취소·환불 상태 전이는 허용된 순서만 통과시킵니다.
- 성공보수와 광고비는 별도 계약·청구 기준을 사용합니다.
- 환불정책과 증빙 발급 기준은 법무·세무 검토 후 확정합니다.
